Dockerfile and docker-compose.yml solve two different problems in containerized applications.

| File                   | Purpose                                                | Think of it as         |
| ---------------------- | ------------------------------------------------------ | ---------------------- |
| **Dockerfile**         | Defines **how to build a Docker image**                | Recipe                 |
| **docker-compose.yml** | Defines **how to run one or more containers together** | Blueprint/Orchestrator |


DOCKERFILE:
A Dockerfile is a set of instructions that tells Docker how to package your application into an image.
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .

RUN pip install -r requirements.txt

COPY . .

CMD ["python", "main.py"]
WHAT HAPPENS: FROM python:3.12-slim->WORKDIR /app->COPY requirements.txt .->RUN pip install -r requirements.txt->COPY . .->CMD ["python", "main.py"]


docker-compose.yml

    Most real applications have multiple services.

    For example, your Multi-Agent RAG system has:

    Chat Service
    Airflow
    Redis
    Qdrant
    PostgreSQL
    API

    Running each manually would be tedious.

    Instead, docker-compose.yml starts all of them together.

    TO RUN: docker compose up

Dockerfile = the recipe for preparing one dish (how to make the application).
Docker Compose = the restaurant manager's plan for running the kitchen (chef, waiter, cashier, dishwasher working together).
Kubernetes (EKS) = the restaurant chain's operations system that manages many kitchens, scales staff based on demand, restarts workers if they fail, and balances customers across locations.

FLOW : for setting qdrant using docker
Check if the container needs to be reached from your host machine or only from other containers, and if from host, expose it with -p HOST_PORT:CONTAINER_PORT.
Check the image's own docs for which ports it actually listens on (Qdrant: 6333 REST/dashboard, 6334 gRPC).
If data must survive restarts, always mount a volume with -v ./local/path:/container/internal/path (Qdrant's internal path is /qdrant/storage).
For one throwaway/standalone service, just use docker run -d --name X -p ... -v ... image — fastest for quick iteration.
Once you have 2+ services that need to talk to each other, move to docker-compose.yml instead of raw docker run.
In compose, each key under services: becomes a hostname other containers can resolve (e.g. vectorizer reaches Qdrant at qdrant:6333, not localhost:6333).
depends_on only controls start order, not "wait until ready" — it doesn't guarantee the dependency has finished initializing.
Use environment: in compose to pass secrets/config (like API keys) instead of hardcoding them into the Dockerfile.
In the Dockerfile, pick a base image that exactly matches your language version (e.g. python:3.13-slim if your lock file was built on 3.13).
COPY dependency manifest files (pyproject.toml, poetry.lock) before COPY of source code, so dependency installs get cached and aren't repeated on every code change.
Install dependencies first, then copy the actual app folders afterward.
End with one CMD per Dockerfile — a single entrypoint representing that container's one job.
If you want to run two different things (e.g. vectorizer and chat app) from one Dockerfile, that's a sign you need two separate services/Dockerfiles instead.
